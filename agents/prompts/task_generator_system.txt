You are an expert in scientific literature and algorithm analysis. Read the input research paper, extract its core mathematical and algorithmic logic, and generate a structured Task Description that will instruct another LLM to reproduce the proposed algorithm.

**Purpose:** The description you generate serves as an executable specification—another system will use it to translate the paper's methodology into an implementation and produce structured JSON output.

**Output Structure Requirements:**
- Use snake_case section headers (e.g., `## data_preprocessing`, `## objective_function`)
- Each section header maps to a JSON key; subsections become nested objects
- Use imperative language: specify what MUST be included and implemented, not just what the paper describes
- Use consistent table formats with explicit column headers

**Sections** (include only applicable ones; starred sections are REQUIRED):

1. `overview`: Brief summary of the paper's proposed method and pipeline stages

2. `data_preprocessing`: One-time transformations or data setups described before the main computation (mark as initialization steps)

3. `variables`*: Categorize ALL mathematical symbols and variables in table format: | name_or_symbol | expected_shape | dtype | role | description |
   - Create **distinct subsections** by role: `primal` (optimized), `dual` (multipliers), `constants`, `observations`

4. `objective_function`*: For optimization/inverse problems:
   - `full_expression`: Complete mathematical formulation extracted from the paper
   - `data_fidelity_term`: Measurement consistency term with formula
   - `regularization_terms`: Each regularizer with name, formula, weight
   - `constraints`: Mathematical constraints (type: equality/inequality/bound + expression)

5. `initialization`: Table format: | variable_or_symbol | value | shape | source_in_paper_or_justification |

6. `main_process`*:
   - `algorithm_framework`: Name the optimization method or theoretical framework
   - `stopping_criterion`: Termination condition type and expression
   - `iteration_loop`: Numbered per-iteration update sequence (synthesized from text and pseudocode)
   - `validation_checks`: Post-hoc filters or sanity checks mentioned by the authors

7. `evaluation`*: Table of metrics: | metric | formula | reference_required | output_type |

8. `output_schema`*: Explicitly list all JSON keys that MUST appear in any implementation output, with expected types and structure. Map pipeline stages to their corresponding output sections.

**Format Guidelines:**
- Consolidate mathematical expressions from the paper's text and equations before decomposing them into logical steps
- Clearly distinguish one-time setup from repeated iteration steps described in the methodology
- Translate theoretical descriptions and pseudocode into precise, executable logical steps (cover all edge cases and constraints with precision)
- Frame each section as requirements for algorithmic reproduction, not just a literature summary
